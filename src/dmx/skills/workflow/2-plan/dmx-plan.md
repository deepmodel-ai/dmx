You are generating the implementation plan for an active ticket. Follow every step in order.

## Step 1 — Resolve the active ticket

If `{{ticket_id}}` was provided, use it.

If not, read `.dmx/activeContext.md` and extract the ticket ID from the `## Active Ticket` section.

If no active ticket is found in either place, stop and tell the user: "No active ticket found. Run `/dmx/create-branch` first, or provide `ticket_id` explicitly."

Store the ticket folder path as `.dmx/tickets/active/{ticket_id}/`.

## Step 2 — Read spec.md

Read `.dmx/tickets/active/{ticket_id}/spec.md`.

If the file does not exist, stop and tell the user: "spec.md not found for {ticket_id}. Run `/dmx/create-branch` to scaffold the spec first."

## Step 3 — Validate Q&A completeness

Locate the `## Questions` section in spec.md. For each numbered question, check whether its `Answer:` line has content beyond whitespace.

If any question has an empty or missing answer, stop and output:

```
spec.md has unanswered questions. Please answer the following before planning:

{list each unanswered question by number and text}

Edit `.dmx/tickets/active/{ticket_id}/spec.md` and re-run /dmx/plan.
```

If all questions are answered (or there are no questions), proceed.

## Step 4 — Analyse the spec

Read the following sections of spec.md:
- `## Context` — understand the problem
- `## Scope` — understand what must be built
- `## Out of Scope` — understand the boundaries
- `## Technical Approach` — understand the design decisions
- `## Questions` — use the answers to resolve any ambiguities

## Step 5 — Generate the phased task plan

Design a set of phases that group logically related work. Each phase should be completable and reviewable independently. Use the following guidance:

**Phase sequencing principles:**
- Data/schema changes come before service layer changes.
- Service layer comes before API/router layer.
- Core logic comes before edge cases and error handling.
- Tests come last (or interleaved if TDD is preferred — use the spec to decide).
- Each phase should produce working, committable code on its own.

**Phase naming:** Use the format `Phase N: {Descriptive Name}` (e.g. `Phase 1: Data Model`, `Phase 2: Service Layer`, `Phase 3: API Endpoints`, `Phase 4: Tests & Validation`).

**Task format:** Each task is a single, concrete action — one file, one function, one migration. Avoid vague tasks like "implement the feature". Prefer: "Add `RateLimitEntry` model to `app/models/rate_limit.py`".

## Step 6 — Write tasks.md

Write the following file to `.dmx/tickets/active/{ticket_id}/tasks.md`:

```markdown
# Tasks: {ticket_id} — {ticket summary from spec.md}

**Spec:** `.dmx/tickets/active/{ticket_id}/spec.md`
**Ticket:** [{ticket_id}](https://{config.atlassian_domain}.atlassian.net/browse/{ticket_id})
**Status:** In Progress

---

## Phase 1: {Phase Name}
- [ ] {Concrete task}
- [ ] {Concrete task}
- [ ] {Concrete task}

## Phase 2: {Phase Name}
- [ ] {Concrete task}
- [ ] {Concrete task}

{...remaining phases}

---
_Generated by /dmx/plan. Edit freely — this is your implementation contract._
```

## Step 7 — Update activeContext.md

Update the `## Active Ticket` section in `.dmx/activeContext.md`:

Change the Tasks line from:
```
- **Tasks:** `.dmx/tickets/active/{ticket_id}/tasks.md` _(not yet generated)_
```
to:
```
- **Tasks:** `.dmx/tickets/active/{ticket_id}/tasks.md` _(N phases, M tasks)_
```

where N and M are the actual counts.

## Step 8 — Display the plan and open the file

Output the full content of the generated `tasks.md` as a markdown block so the developer can review it in-chat.

Then output:

```
Plan saved: .dmx/tickets/active/{ticket_id}/tasks.md
{N} phases, {M} tasks total.

Open the file and press Cmd+Shift+V (Mac) / Ctrl+Shift+V (Windows) to preview.

When ready, run /dmx/implement-next-phase to begin Phase 1,
or /dmx/implement-next-task to step through tasks one at a time.
```

## Guards

- Do not generate tasks.md if any question in spec.md is unanswered. The spec is the contract.
- If tasks.md already exists, warn the user: "tasks.md already exists for {ticket_id}. Regenerating will overwrite the existing plan including any checked-off tasks. Proceed? (yes/no)" — wait for confirmation before overwriting.
- Never generate a phase with a single vague task. If you cannot produce ≥ 2 concrete tasks for a phase, merge it into an adjacent phase.
